# Язык структурированных запросов

---

Вы работник компании, занимающейся изобретением велосипедов. Для учёта моделей велосипедов компании необходимо
организовать базу данных. Вам поступила задача разработать СУБД и модель данных... Чтож, кто вы такой, чтобы спорить с
начальством?

## 1. Разработать формальную грамматику языка структурированных запросов

Разрабатываемый язык является регистро**НЕ**зависимым.
Язык должен поддерживать следующие литералы:

- целочисленные константы - совокупность цифр. Допускается разделение разрядов символом "\_", если он стоит не на границе
  числа (например, `1_000_000`);
- вещественные константы - совокупность цифр и символа ".". Допускается разделение разрядов символом "\_", если он стоит
  не на границе числа и не стоит рядом с ".";
- строковая константа - любой текст, заключенный между символами двойных кавычек `"`, символ экранирования `\`;
- логическая константы - терминалы `TRUE` и `FALSE`;
- указатель отсутствия данных - терминал `NULL`;
- имя - набор букв, цифр и символов `[_$]`, не начинающийся с цифры.

### Подмножество языка для управления базами данных и таблицами

Подмножество языка, отвечающее за управление базами данных и таблицами, должно поддерживать следующие конструкции:

- `CREATE DATABASE <имя базы данных>;`, предложение создания базы данных. Строковый литерал указывает название базы
  данных;
- `DROP DATABASE <имя базы данных>;`, предложение удаления базы данных и всех её таблиц. строковый литерал
  указывает название базы данных;
- ```
  CREATE TABLE <имя таблицы> (<список столбцов>) [WITH INDEXING BY <тип индексации>];
  ```
  предложение создания новой таблицы.  
  Тип индексации задает то, как таблица хранится в памяти. Может принимать значения `ORDER`, `HASH` и `TREE` (функционал индексации опционален).
  Подробнее об индексации ниже.
- `<имя> <тип данных> <модифицаторы, разделенные пробелами>` - описание столбца таблицы. Доступные модификаторы
  столбца и типы данных описаны ниже;
- `ALTER TABLE <имя таблицы> ADD CONSTRAINT <имя ограничения> <тип ограничения> (<столбцы>);` - описание простых
  именованных
  ограничений. Типы ограничений приведены ниже.
- `ALTER TABLE <имя таблицы> ADD FOREIGN KEY <имя ограничения> (<столбцы>) TO <имя таблицы>(<список ключей>) ;` -
  описание
  внешних ключей. Перечисленные столбцы данной таблицы ставятся в соответствие перечисленным столбцам другой таблицы.
  При этом во второй таблице эти столбцы должны или быть первичным ключём, или на них должно быть ограничение UNIQUE;
- `ALTER TABLE <имя таблоицы> ADD COLUMN <имя> <тип данных> <модифицаторы, разделенные пробелами>` - добавление нового
  столбца в таблицу;
- `ALTER TABLE <имя таблоицы> EDIT COLUMN <имя> <тип данных> <модифицаторы, разделенные пробелами>;` - изменение столбца
  в таблице;
- `ALTER TABLE <имя таблоицы> DROP COLUMN <имя>;` - удаление столбца из таблицы;
- `DROP CONSTRAINT <имя ограничения>;` - удалить именованное ограничение;
- `DROP TABLE <имя таблицы>;` - удаление таблицы из базы данных.

Для таблиц поддерживаются следующие типы индексации:

- `ORDER` - таблица хранится как отсортированный по ключу вектор. Выборка по ключу осуществляется с помощью
  бинарного поиска;
- `HASH` - таблица хранится на хэш таблице, выборка по ключу осуществляется за O(1);
- `TREE` - таблица хранится на дереве, выборка по ключу осуществляется за O(log n);
- без индексации. таблица хранится на векторе, соблюдение порядка не гарантируется. Поиск за O(n).

Невозможно создать индексированную таблицу, если в ней не определен первичный ключ.

Столбцы могут иметь следующие типы данных:

- `TEXT` - строковые литералы и `NULL`;
- `LOGIC` - логические значения (`TRUE`, `FALSE` и `NULL`);
- `INTEGER` - целочисленные значения и `NULL`;
- `REAL` - вещественные значения и `NULL`.

Столбцы могут иметь следующие модификаторы:

- `PRIMARY KEY` - столбец является первичным ключом;
- `UNIQUE` - значение данного столбца не могут повторяться в рамках таблицы;
- `NOT NULL` - значение данного столбца не могут быть null.

К простым ограничениям целостности относятся:

- `UNIQUE` - совокупная уникальность столбцов. Строки конфликтуют только если ВСЕ перечисленные столбцы одновременно
  имеют одинаковые значения;
- `PRIMARY KEY` - создание составного первичного ключа. Если в таблице уже задан первичный ключ, то данная операция
  возвращает ошибку;

### Подмножество языка для осуществления запросов

Запрос на получение строк таблицы имеет следующий вид:

```
SELECT <проекция>
FROM <таблица> [[LEFT | RIGHT | FULL] JOIN <таблица> ON <список столбцов>]
[WHERE <условие>]
[ORDER BY [ASC | DESC] столбец[, ...]]
[SKIP <число>] [LIMIT <число>]
[GROUP BY <список столбцов>];
```

Условие - выражение, которое возвращает логическое значение.  
Под таблицами подразумевается или имя таблицы, или SELECT выражение в скобках. То есть, например, предложение
`SELECT a FROM (SELECT * FROM tbl);` синтаксически корректно.

Проекция это список столбцов, которые получатся в той таблице, что формирует SELECT. Элементы этого списка или просто
имена столбцов из таблицы в блоке FROM, или же выражение вида `<выражение> AS <имя>`. Имя задает то, как будет
называться столбец в формируемой выборке. Так же допускается написать просто `*`, это значит, что в результирующей
таблице столбцы такие, как были в исходной.

Выражение может содержать литералы, арифметические операторы `+`, `-` ,`*` ,`/` и `%`, логические операторы `AND`, `OR`,
`NOT`, операторы сравнения (равенство `=`, неравенство `<>`), ссылки на столбцы таблицы. Если имеет место конфликт имён столбцов,
используется конструкция вида `<имя таблицы>.<имя столбца>`.

Выражения так же могут содержать подзапросы, валидация типа данных результата выполнения подзапроса выполняется во время
исполнения.
Например, допустимо писать выражение вида `a = (SELECT c FROM tbl WHERE b = 1)`. Здесь мы из таблицы
`tbl` получили строки, в которых столбец b равен 1, из этих строк взяли только столбец "c", выражение проверяет,
что в текущем отношении столбец a равен найденному значению.

`JOIN` задает сцепление двух таблиц. По-умолчанию это внутреннее сцепление. `LEFT JOIN` - левое сцепление,
`RIGHT JOIN` - правое сцепление, `FULL JOIN` - двустороннее (полное) сцепление.
Сцепление производится между строками, если в обеих таблицах совокупность перечисленных столбцов равна.

`WHERE` задает условие, по которому отбираются строки. В итоговую таблицу войдут только те строки, что подходят под
условие.

`ORDER BY` задает сортировку строк в таблице. `ASC` - от меньшего к большему, `DESC` - от большего к меньшему. Если
указано несколько столбцов, то сортировка по следующим выполняется при равенстве всех предыдущих.

`SKIP` - удалить первые N столбцов.
`LIMIT` - оставить только N столбцов.

`GROUP BY` - группировка строк, в которых заданные столбцы совокупно равны. Если используется блок `GROUP BY`, то все
столбцы, которые не попали в него, могут присутствовать в проекции только внутри агрегирующих функций.

Доступны следующие агрегирующие функции:

- `min(<выражение>)`/`max(<выражение>)` - самый маленький/большой элемент в группе. NULL, если все элементы в группе
  NULL;
- `sum(<выражение>)` - сумма всех элементов в группе. NULL игнорируются. NULL, если все элементы в группе NULL;
- `count()` - размер группы;
- `reduce(<выражение>, <выражение>)` (опционально) - произвольная свертка. Первое выражение задает начальное значение,
  второе значение задаёт правило накопления, в этом выражении можно сослаться на текущий накопленный результат
  `current`. Например, накопление произведения элементов столбца `a` можно будет задать так:
  `prodA = reduce(1, current * a)`

Запрос на вставку имеет следующий вид: `INSERT INTO <имя таблицы> VALUE (<список выражений>);`.

Запрос на удаление имеет следующий вид: `DELETE FROM <имя таблицы> [WHERE <условие>]`.

Так же в контексте выражений определены глобальные функции, на которые можно ссылаться.
Обязательно к реализации функция `matches(<regex>, <выражение>)`. Она возвращает true, если заданная
выражением строка удовлетворяет регулярному выражению, заданному строковым литералом.  
(Если хотите, можете использовать свою вторую лабу, но допускается и стандартная библиотека регулярных выражений)

## 2. Разработать интерпретатор запросов к СУБД

Разработать Систему Управления Базами Данных (СУБД). Система состоит из сервера, обрабатывающего запросы, используя на
транспортном уровне TCP, и, опционально, клиента, который позволяет отправлять запросы, введенные пользователем с
клавиатуры.

Система должна поддерживать следующие операции:

1. Создание/удаление базы данных;
2. Создание/удаление отношений базы данных;
3. Добавление/изменение/удаление столбцов в таблицых;
4. Запросы получения и изменения информации.

Запросы, которые подразумевают работу с конкретной базой данных, должны содержать имя базы данных в URL.  
Допустимо использовать HTTP протокол с передачей запроса как `plain text`. В этом случае для демонстрации можно
использовать Postman или SOAP UI.

Пример возможных ЭП:

- `http://host:port/` - обрабатываются только запросы создания и удаления базы данных;
- `http://host:port/{database}` - обрабатываются все запросы.

(Опционально) - создать свой протокол, работающий на TCP сокетах, чтобы в URL можно было писать заголовок не "http://",
а "mydb://" или что-то ещё.

# 3. Разработать модель данных для учёта моделей велосипедов

Используя разработанную СУБД разработать модель данных. Имеются следующие сущности:

Компонент велосипеда - id, тип компонента (руль, рама, колесо, седло и т.д.), название, описание компонента, стоимость
компонента;  
Велосипед - id, название, описание, список компонентов, из которых состоит (можно указать количество данного компонента
в велосипеде).

Составить запрос, который позволяет получить стоимость велосипеда (как сумму стоимости компонентов велосипеда, с учетом
кратности их использования).
